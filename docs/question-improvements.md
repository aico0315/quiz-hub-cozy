# 問題データの改善まとめ

## 1. 問題文の区切り文字を明示（l-8）

### 何が起きたか

パンくずリスト問題で、正しいコードを書いているのに不正解になるケースがありました。

```
あなたの出力：  "メンズ/トップス/セール"
期待する出力：  "メンズ / トップス / セール"
```

### 原因

問題文に `` ` / ` `` と書いていましたが、バッククォートの中のスペースはスマホ画面では
見落としやすく、スペースなしの `'/'` で書いてしまうと不正解になっていました。

### 修正

区切り文字にスペースが含まれることを問題文に明示しました。

```
修正前：カテゴリの配列から ` / ` で繋いだ文字列にして
修正後：カテゴリの配列から `/`（スラッシュの前後にスペースあり）で繋いだ文字列にして
```

### 教訓

**問題文の曖昧さはユーザーのミスではなく設計のミス**です。
初学者向けのアプリでは「見て当然」ではなく「書いてあって当然」を意識しましょう。

## 2. 浮動小数点誤差への対処（l-2）

### 何が起きたか

税込み価格を計算する問題で、正しそうなコードが不正解になりました。

```js
// このコードだと...
prices.map(p => p * 1.1)
// 3000 * 1.1 = 3300.0000000000005 になってしまう
```

### 原因

JavaScriptは数値をIEEE 754という方式で内部的に管理しているため、
一部の小数計算で微小な誤差が生じます。これはJavaScriptだけでなく
ほとんどのプログラミング言語で起きる仕様です。

### 修正

`Math.round()` で丸めることで正確な整数が得られます。
解説に浮動小数点誤差の説明と `Math.round()` を使う理由を追記しました。

```js
// 正しい書き方
const result = prices.map(p => Math.round(p * 1.1));
```

## 3. 解説への実務的な補足追加

正解を導くだけでなく「実務ではこう書く」という視点を解説に追加しました。

### l-3（sort）: 破壊的メソッドの注意

`sort()` は元の配列を直接書き換えます。実務では元データを残したいケースが多いので、
スプレッド構文でコピーしてから並び替えるのが安全です。

```js
// NG: 元の配列が変わってしまう
sales.sort((a, b) => a - b);

// OK: コピーしてから並び替える
[...sales].sort((a, b) => a - b);
```

### l-6（Math.max）: 大量データへの対応

スプレッド構文は配列が非常に大きい場合（数万件以上）にエラーになることがあります。
大量データを扱う実務では `reduce` の方が安全です。

```js
// 数万件だとスタックオーバーフローの恐れ
Math.max(...pageViews);

// 安全な代替
pageViews.reduce((max, v) => v > max ? v : max, -Infinity);
```

### l-10（三項演算子）: 条件の意図を明確に

`stock >= 1` より `stock > 0` の方が意図が明確です。
返品処理などでマイナス在庫が発生するシステムでは特に重要です。

また、条件分岐が複数になる場合は早期リターンの方が読みやすくなります。

```js
// 早期リターンの例
if (stock <= 0) {
  console.log('在庫なし');
  return;
}
console.log('在庫あり');
```

### l-13（async/await）: エラーハンドリングは必須

`try/catch` なしの非同期処理は実務では NG です。
APIが失敗したときに何も対処しないと、アプリが止まったりユーザーに何も表示されなくなります。

```js
// 実務での書き方
async function main() {
  try {
    const product = await fetchProduct();
    console.log(product.name);
  } catch (error) {
    console.error('取得に失敗しました:', error);
  }
}
main();
```
